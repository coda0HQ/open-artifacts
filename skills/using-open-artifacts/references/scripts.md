# Runtime libraries (self-hosted, same-origin)

This is the reference for the runtime-library surface — currently just
[mermaid](https://mermaid.js.org/). It sits beside `design.md`'s visual-material
guidance (SVG diagrams/charts as the CSP-clean default). Reach for an inline
SVG first; reach for mermaid when a diagram needs many variants the author will
iterate on, or when the source is authored as text and rendering to SVG by hand
is impractical.

**Why same-origin, not CDN-direct?** The mermaid bundle is vendored into
`public/vendor/mermaid.bundle.mjs` (regenerated by `scripts/vendor-mermaid.mjs`
in the open-artifacts repo) and served same-origin from
`GET /vendor/mermaid.bundle.mjs`. The browser loads it under
`script-src 'self'` — **no external script host appears in the CSP**, so an
artifact's inline JS cannot `createElement("script", {src: <external>})` to
pull an arbitrary package: there is no host to allowlist. That closes the
inline-JS bypass the jsdelivr allowlist carried, without breaking user JS (the
per-request nonce is stamped on every user `<script>` at serve time). A
Worker-side proxy would add a maintenance surface for nothing; the static asset
is cached and immutable, and the build gate (not a proxy) enforces which bundle
may load.

## The contract

Declare the library by writing one `<script type="module" src>` tag in the body
fragment, pointing at the same-origin vendor path:

```html
<script type="module" src="/vendor/mermaid.bundle.mjs"></script>
<pre class="mermaid">
flowchart LR
  A[Author] --> B(Build gate)
  B --> C{same-origin /vendor + allowlisted?}
  C -->|yes| D[(browser loads from /vendor)]
  C -->|no| E[rejected]
</pre>
```

The bundle is an ESM that, on load, assigns `window.mermaid` as a side effect
(appended at vendor time), so the plain inline JS in the scripts slot can reach
it. Initialize from the scripts slot (`document.fragments.scripts`):

```js
// mermaid is attached to window by the /vendor bundle; no import needed.
if (window.mermaid) {
  window.mermaid.initialize({ startOnLoad: true });
  // or, after the diagrams: await window.mermaid.run({ querySelector: ".mermaid" });
}
```

`mermaid.initialize` / `mermaid.run` both go in the scripts slot — **not** in
the body. Inline `<script>` (no `src`) is still forbidden in the body.

**URL grammar:** `<script type="module" src="/vendor/<bundle>.mjs">`

- `bundle` — an allowlisted vendor filename. Today: `mermaid.bundle.mjs`.
- The path must be the exact same-origin `/vendor/<bundle>.mjs` form.

The only accepted form is `/vendor/mermaid.bundle.mjs`. The browser fetches the
bundle same-origin (the CSP keeps `script-src 'self'`); there is no external
script host. The build gate restricts the URL to the allowlisted bundle, so an
artifact cannot reference any other path.

## Opt-in caveat (read first)

The runtime-library *bundle* is always served — mermaid is a build-time-
allowlisted library, so the route and the static asset ship on every deploy.
What is opt-in is the web-font surface (the `OPEN_ARTIFACTS_WEB_FONTS` env var),
which lifts the sandbox's opaque origin (`allow-same-origin`) so the browser can
cache fonts. On a deploy that has not opted in, the sandbox keeps its opaque
origin and the mermaid bundle still loads same-origin.

The `OPEN_ARTIFACTS_WEB_FONTS` flag's trade-off (documented in `design.md`'s
Hard constraints) is real for fonts (it adds `allow-same-origin`, so a malicious
artifact on such a deploy can read the host origin's `localStorage`/cookies).
For scripts there is **no** external-code-execution surface: the only loaded
library is the vendored mermaid bundle, served same-origin, and the build gate
restricts which bundle path an artifact may name.

## Discipline

- **One library, pinned.** Mermaid is ~3.5 MB. Don't load it for a single small
  diagram — hand-draw the SVG (see `design.md`'s visual-material section). Load
  it when there are many diagrams or the source is text-authored.
- **Initialize once.** Call `mermaid.initialize`/`mermaid.run` from the scripts
  slot, after the `<script type="module" src>` tag in document order. The
  /vendor bundle attaches `window.mermaid` on load.
- **Theme the diagrams.** Mermaid reads CSS variables for theming; set
  `--mermaid-*` tokens (or pass `themeVariables` to `initialize`) in both the
  light and dark `:root` blocks so diagrams match the direction. Default mermaid
  theme is light-only and will look wrong in a dark block.
- **Keep the fallback honest.** If the deploy has blocked the /vendor route or
  the bundle is absent, `window.mermaid` is undefined — guard the init call
  (`if (window.mermaid) ...`) so the page degrades to the raw `<pre>` source
  rather than throwing.
- **Syntax is validated at build time.** Every `<pre class="mermaid">` is
  parsed by the vendored mermaid + linkedom bundle during `validate`/`publish`;
  a syntax error fails the build with mermaid's message (line number + expected
  token). So a broken diagram never ships — fix the source and re-validate. If
  the `vendor/mermaid/` bundles are absent (skill installed before this gate),
  the gate warns and skips rather than blocking; run
  `node scripts/vendor-mermaid.mjs` in the open-artifacts repo to (re)generate.

## Adding a bundle

The allowlist is intentional, not a config file. To add a library:

1. Vendor the bundle into `public/vendor/<name>.bundle.mjs` (a browser-loadable
   ESM that attaches its API to `window` as a side effect), via a script in
   `scripts/` modeled on `vendor-mermaid.mjs`.
2. Add the `/vendor/<name>.bundle.mjs` path to `ALLOWED_SCRIPT_BUNDLES` in
   `skills/using-open-artifacts/scripts/lib/validate.mjs`.
3. Add a Worker route (or rely on static serving) so the bundle is served
   same-origin with `text/javascript` + `nosniff`.
4. Document the canonical path here.
