Artifacts on Cloudflare.

An agent skill plus a Cloudflare Worker (Hono + D1 + R2) that lets any coding agent publish self-contained HTML/Markdown pages to shareable URLs, protect them with zero-knowledge passwords, and keep them updated as the project evolves. Runs entirely on Cloudflare, fits the free tier, no accounts anywhere.

What it is

An open-source clone of Claude Code Artifacts. The skill composes body-only HTML from a Recipe; the Worker wraps it in a sandboxed skeleton; D1 holds metadata and token hashes, R2 holds content bodies — both strongly consistent.

identityNo accounts. 12-char crypto-random ids. Creation returns a one-time write token; only its SHA-256 is stored.
channelsartifact.channel binds a stable URL; the CLI keeps a per-channel token (ch_) in credentials. Only the hash is stored.
versionsEvery publish is an immutable version with its own title/description/favicon/format/encryption state. ?v=N views history; baseVersion mismatch → 409.
markdownVendored marked inlined — encrypted and plain share one client-side code path.
passwordsPBKDF2-HMAC-SHA256 (600k iterations) + AES-256-GCM, client-side. Server stores only {salt, iv, ciphertext}.

How publishing works

A zero-dep Node CLI (artifact.mjs) is the agent's only tool. Every artifact is generated from a versioned JSON Recipe plus ordered fragments; the CLI validates and composes in memory, then sends one final publish request. Run validate/create from the project root with a project-relative Recipe path — an absolute path or a non-root cwd fails with recipe must live inside the project root.

node skills/using-open-artifacts/scripts/artifact.mjs validate artifacts/app.recipe.json node skills/using-open-artifacts/scripts/artifact.mjs create artifacts/app.recipe.json
  1. Recipe — JSON owns title, favicon, scope, watch globs, channel, level, Canvas mode, locality, and encryption policy. Fragments are ordered theme, styles, body, scripts.
  2. validate — local builder checks path traversal, duplicate or oversized fragments, external scripts/styles, remote media, network APIs, and (for Canvas) frame IDs, geometry, connectors, tours, and builder-owned controls. Writes nothing.
  3. build — composes final bytes in memory: injects tokens.css, the theme fragment, then vendored Canvas CSS/JS when canvas: true. No timestamps or random values — the build is deterministic.
  4. create — sends one request; the server returns id, a one-time writeToken, and a URL. Manifest v2 records the Recipe path and Recipe/input/output hashes only after the publish succeeds.
  5. update — same id, new immutable version; reads the Recipe from Manifest v2, rebuilds and re-validates in memory, publishes once. --label is capped at 60 bytes (CJK chars are 3 bytes each — keep labels terse ASCII).
  6. status — hashes watched files against the last snapshot, reports every stale artifact; ack <id> advances the baseline without republishing when the change is out of scope.

Levels & canvas

level sets fidelity and motion budget. canvas swaps the shell — orthogonal to level.

L1
Typography-led document. No flashy hero, no JS state.
L2
Information design. Scannable, stateful, no external requests.
L3
Editorial / showcase. One orchestrated motion moment, native APIs only.

canvas: true swaps a scrolling document for a pan/zoom plane of frames. Composes with any level: L1 spatial notes, L2 multi-frame prototype (the default), L3 canvas-as-showcase.

This board is level 3, canvas: true — the composition is the hero, the overview→focus zoom is the one orchestrated moment.

Validation gates

The local builder enforces these before publish. All are static; the strict CSP makes some of them structural.

body <style>/<script>HTML body fragments cannot contain document wrappers, <style>, or <script> — CSS goes in fragments.styles, JS in fragments.scripts, composed by the builder.
container-classIf CSS defines a .class with max-width, the body must apply it (statically or via a quoted JS token) — catches the silent defect where a cap is defined but never used and the page ships at 100% width.
contrast P0Resolves authored --fg/--bg/--muted/--surface/--accent overrides (hex/oklch/rgb/hsl literals only) and fails any pair under 4.5:1 in either theme. Contract defaults already pass.
measure cap (L1)A level-1 non-canvas HTML doc must constrain body width (.oa-prose or a max-width) or the build fails — the default is a full-width doc with browser spacing. Canvas and L2/L3 are exempt.
scrollspy last-tabDetect-only: if a nav scrollspy ships the last-tab-stays-inactive-at-scroll-bottom bug (tight IO rootMargin and no scroll-boundary fallback and scrollIntoView inside setActive), it fails. No scrollspy → gate silent.
canvas ABIExactly one #canvas + one #plane nested inside it; every frame a <section> with kebab id, --x/--y/--w/--h, one button.oa-frame-label[type=button], one inert div.oa-frame-body; no authored #zoom-*/#tour-* (builder-owned).
dup style=A start tag carrying style= twice fails — HTML parsers keep only the first, silently dropping the later value.

Viewer chrome & sandbox

sticky headerA service header (favicon + title, brand chip, theme toggle) is pinned to the top by the harness at serve time — never hand-authored. Its measured height is exposed as --oa-header-h for sticky offsets.
theme toggleStamps data-theme="light|dark" on <html>; both theme blocks are mandatory and must win over prefers-color-scheme in both directions.
opaque originCSP sandbox allow-scripts ...; default-src 'none' gives the document an opaque origin — no cookies, no localStorage, no same-origin API calls, no fetch/XHR/WebSocket.
no storageKeep state in memory (JS variables) for the session. The theme toggle wraps localStorage in try/catch (opaque origin throws) — theme choice is per-load.
LAYOUT_SCRIPTThe harness rewrites top of every position: sticky body descendant to var(--oa-header-h) — so canvas floating controls must be position: fixed, never sticky.
OG cardEvery page emits OpenGraph + Twitter tags; /og/:id returns a 1200×630 PNG rasterized on the edge with embedded Inter — no external requests.

Storage & bindings

BindingRole
DB (D1)Metadata, token hashes, version index. Strongly consistent (read-after-write cross-colo). D1's 2 MB row cap forbids storing HTML here.
CONTENT (R2)Content bodies at content/<id>/<version>. Strongly consistent via bindings. Per-version customMetadata.encrypted flag — an artifact can switch encrypted/plain across versions.
ASSETSStatic assets dir (public/) for the landing page. run_worker_first routes /, /api/*, /a/*, /og/* through the Worker first.
4 MiB capMax artifact content size is 4 MiB (post-base64 for encrypted — encrypted max plaintext ≈ 2.97 MiB). Fits the free-tier envelope.
schema autoD1 schema applies itself on first request (CREATE TABLE IF NOT EXISTS, memoized per isolate) — zero-step deploys. Lazy ALTERs add columns to existing DBs in place.
compare-and-swapUpdates are safe: D1 advances current_version only if it still matches the snapshot, then R2 writes — an R2 object is never orphaned from D1. Concurrent PUTs: exactly one wins, the rest get 409.
rate limitsBest-effort, per-colo: create 20/min/IP, update 60/min/id. Free tier headroom: Workers 100k req/day, R2 1M writes/mo, D1 100k row writes/day.

Local vs shared recipes

Fragment paths resolve against the Recipe file's own directory, not cwd. Shared Recipes/fragments are commit-ready reproducible sources; local or encrypted sources live under gitignored .artifacts/.

.artifacts/ recipes.local/ report.recipe.json # paths use ../fragments.local/ fragments.local/ report/ theme.css body.html behavior.js
sharedRecipe JSON + fragments in the project tree (e.g. artifacts/ or .artifacts/recipes/). Commit-ready. This board's Recipe lives at .artifacts/recipes/open-artifacts-canvas-overview-pg7zuNun.recipe.json.
localartifact.local: true places the Recipe under .artifacts/recipes.local/ and fragments under .artifacts/fragments.local/, with state in manifest.local.json. A misplaced local Recipe reports both rules at once.
encryptedAlways local: names a passwordCredential, resolved from --password, an OPEN_ARTIFACTS_PASSWORD_* env var, or credentials.json. Passwords never enter Recipes or Manifests.
path ruleRun validate/create from the project root with a project-relative Recipe path. An absolute path, or one whose real location resolves outside cwd, fails with recipe must live inside the project root.
